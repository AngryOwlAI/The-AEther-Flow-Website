# Project Explainer Frontend Output Contract

The generated frontend must explain The AEther Flow Project without becoming
the authority for its claims.

## Required Sections

For a single-page explainer, include:

1. Hero: project or page name, concise thesis, and one reading action.
2. What it is: plain-language explanation of the page subject.
3. Why it exists: problem, reader need, or research gap.
4. How it works: high-level architecture, workflow, or source relationship.
5. Evidence and state: claim status, limitations, and source references.
6. Reading path: important pages, resources, or source-backed artifacts.
7. Next action: review, read, download, or follow the next approved surface.

## Claim Boundaries

Distinguish:

- Existing website functionality.
- Reviewed upstream claims.
- Draft/control material.
- Proposal-only material.
- Human-gated or unverified claims.
- Known limitations and blocked downstream claims.

## Forbidden Behavior

Do not:

- Invent scientific, mathematical, governance, or research-workflow claims.
- Present the website as a runtime UI unless explicitly asked and supported.
- Add fake dashboards, metrics, simulations, proof validation, or model output.
- Use placeholder filler.
- Leak local source paths in built public output.

## Accessibility Minimum

- One `h1` per page.
- Semantic landmarks where practical.
- Meaningful link text.
- Alt text for informative images.
- Visible focus states.
- Body text contrast near WCAG AA.
- Mobile layout checked at narrow width.
- Reduced-motion handling if motion is added.
