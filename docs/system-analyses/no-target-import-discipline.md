# No-Target-Import Discipline System Analysis

## Purpose

This analysis supports PG-011: creating
`/project/physics/no-target-import-discipline/` to explain why AEther Flow
blocks hidden target-metric, target-topology, target-atlas, proper-time,
detector-semantics, benchmark-success, and process-authority imports.

The route is methodological. It explains a guardrail, not a proof of any
candidate and not a guarantee that all target imports have been eliminated
from all future work.

## Scope And Authority

This document is website-maintained explanatory analysis. It is not source
authority and does not change research state. The upstream guard map and
criterion artifacts remain authoritative for the research-control record.

The no-target-import discipline is a fail-closed source-side filter. It asks
whether a candidate proof or construction can factor through allowed source
data rather than hidden target context or process metadata. Passing a scoped
audit means only that no prohibited import was detected under the stated
scope. It does not adopt `M_src`, define `g_eff`, derive matter coupling,
derive Einstein equations, promote the benchmark, issue Gate Chair closure, or
complete GR derivation.

## System Context

No-target-import discipline exists because a derivation from source structure
would be invalid if it quietly borrows the target geometry it is supposed to
recover. The same problem appears in process form: a registry row, validator
PASS, role identity, handoff, generated derivative, local cache, file order, or
commit timestamp can document a transaction, but none of those surfaces is a
mathematical proof.

The formal criterion introduces a diagnostic target context `T`, including
target topology, target atlas, target metric, proper-time semantics, detector
semantics, benchmark success data, and process metadata. A source-only
construction should remain invariant when that target context is replaced.
Equivalently, each non-bottom proof step should factor through source data
`S(E)` rather than target context.

## Functionality Or Topic Analysis

### Forbidden import classes

- Target topology: target open sets, neighborhoods, or manifold topology.
- Target smooth atlas: target coordinate charts, differentiability class, or
  smooth transition maps.
- Target metric: metric tensor, Lorentzian signature, causal cones, line
  element, distance, or proper time.
- Empirical observer protocol: detector response, clocks, or empirical
  observer semantics as proof inputs.
- Benchmark success: exact-GR recovery or benchmark behavior used as source
  evidence.
- Process authority: validator, registry, role, handoff, generated derivative,
  local cache, file order, commit status, or recency used as proof.

### Acceptable source-side construction

Acceptable construction may use explicitly licensed source data, source
supports, source overlap predicates, source transition tokens, source
certificates, and bottom guards. It must preserve the distinction between
source support names and target coordinate charts, between source transition
tokens and target smooth transition maps, and between finite/local evidence
and global adoption.

### Formal criterion

The Phase 5 criterion can be summarized as:

`C_src(E; T) = F_src(E)`

where `T` is a diagnostic target context and `F_src(E)` is source-only. If
object equality is too strong for a later formal setting, the replacement is
proof-step factorization: every non-bottom assertion must be expressible
without using target or process context.

Criterion status remains `criterion_formalized_pending_audit`. It narrows the
burden and makes audits more precise, but it does not discharge the adoption
theorem.

## Risks, Failure Modes, Claim Boundaries

Primary risk: readers may interpret a guardrail as proof that target import has
been eliminated everywhere. The route must reject that.

Safe claims:

- No-target-import discipline blocks hidden target and process authority from
  entering source-side proofs.
- The guard map is a reusable routing and validation checklist.
- The criterion artifact formalizes a source-only factorization test.
- The criterion status is `criterion_formalized_pending_audit`.
- A scoped audit pass would mean no prohibited import was found under the
  stated scope, not adoption.

Forbidden implications:

- All future work is target-import-free.
- `M_src` has been adopted.
- A physical metric, `g_eff`, matter coupling, Einstein equations, benchmark
  promotion, or completed derivation has been produced.
- Validator, registry, role, handoff, generated derivative, local cache, file
  order, commit status, or recency is mathematical proof.
- Benchmark success can be used as source evidence for a source construction.
